## Getting Started

This project is a browser extension built with [WXT](https://wxt.dev), a meta-framework that scaffolds and builds extensions for Chrome, Firefox, and other Chromium-based browsers.

### Prerequisites

-   **Node.js v18 or higher** (developed against v24). Verify with:

    ``` bash
    node --version
    ```

-   A Chromium-based browser for testing (Chrome recommended).

### One-time setup

From the repo root:

``` bash
cd extension
npm install
```

### Running in development mode

``` bash
npm run dev
```

This launches a fresh Chrome instance with the extension pre-loaded. The Chrome window uses an isolated, temporary profile — it won't touch your normal browser data.

The dev server watches your files and auto-reloads on most changes. **Exceptions:** - Changes to `wxt.config.ts` require a manual restart (`Ctrl+C`, then `npm run dev` again). - Content script changes occasionally fail to hot-reload — restart if behavior seems incorrect.

### Project structure

```         
extension/
├── entrypoints/         # Each file here is a "door" the browser uses to enter the extension
│   ├── background.ts    # Background service worker — registers context menu, dispatches messages
│   ├── content.ts       # Content script — runs inside web pages, finds images, draws overlay
│   └── popup/           # Default WXT scaffold popup (currently unused by our feature)
├── wxt.config.ts        # WXT config — declares manifest fields, permissions, etc.
├── package.json         # Project dependencies and npm scripts
└── .output/             # Build output (gitignored) — generated by `npm run dev` or `npm run build`
```

WXT generates the extension's `manifest.json` automatically from the project structure plus `wxt.config.ts`. You normally don't edit the manifest by hand.

### Testing the current MVP feature

1.  Run `npm run dev`.
2.  In the auto-launched Chrome window, navigate to any page with images.
3.  Right-click any image → **Validate image**.
4.  After a brief delay (\~500ms), a green checkmark should appear in the image's top-right corner (ideally, but correct placement was hit-or-miss during recent testing).

The validation is currently a **stub** — it always succeeds after a fake delay. Replacing it with the GeoCam validation service is the next step (see below).

### Useful debugging tools

-   **Service worker console** (background script): `chrome://extensions` → toggle on Developer mode → find this extension → click "service worker" under "Inspect views".
-   **Page console** (content script): right-click any page → Inspect → Console tab.

The two scripts log to *different* consoles, since they run in different contexts. Open both when debugging.

## Next Step: Replace the Stub With GeoCam Validation

The current MVP uses a **stub** validation function (`stubValidate` in `entrypoints/content.ts`) that fakes a 500ms delay and always returns `true`.

### About GeoCam validation

The GeoCam backend is a separate Python project (with Docker support) that exposes an HTTP API. It is not currently hosted, so it must be run locally during development.

### What needs to change in the MVP extension (to apply once the GeoCam API is understood)

Independent of the GeoCam API specifics, the following design choices are worth keeping in mind:

#### Where the HTTP call should live

Two reasonable options:

-   **Option A: Call directly from the content script.** Simpler. Replace `stubValidate` directly with a real call.
-   **Option B: Call from the background script.** Cleaner separation: the content script messages the background, the background calls the API, and replies with the result. This is closer to a "production" architecture and avoids potential CORS issues (background scripts can call any URL the manifest permits, regardless of the page's CORS policy).

**Option B is recommended** for anything beyond a quick test. Reasons: - Avoids potential CORS issues (background scripts are exempt from many CORS restrictions that content scripts face). - Keeps secrets/keys out of the content script's context, which shares its environment with the host webpage. - Centralizes the integration so future plugins can reuse the same logic.

#### Manifest permissions

Whichever script makes the HTTP call, the extension must explicitly declare which external hosts it's allowed to contact. In `wxt.config.ts`:

``` typescript
export default defineConfig({
  manifest: {
    permissions: ['contextMenus'],
    host_permissions: ['http://localhost:*/*'],  // adjust to actual host
  },
});
```

If the extension stores keys, you'll also need the `'storage'` permission.

#### Stub-first refactor

A safer integration order than immediately deleting the stub:

1.  Move `stubValidate` to its own module (e.g., `utils/validate.ts`) so it has a clean import point.
2.  Implement the real version (`realValidate`) alongside it, behind a feature flag or environment check.
3.  Once `realValidate` works against a running local service, retire the stub.

Toggle between stub and real validation while debugging — useful when the service is misbehaving and you want to confirm the extension itself is fine.

### Verification (once integrated)

-   Start the validation service locally.
-   Run `npm run dev`.
-   Right-click an image → Validate image.
-   Confirm the checkmark appears for images the service deems valid, and does *not* appear for invalid ones.
-   Check both consoles (page and service worker) for any errors.

### Considerations once basic integration works

-   **Error handling**: What should happen if the service is unreachable? Show an error overlay? Silent fail? Currently the code falls back to "do nothing."
-   **Loading state**: Image validation may take longer than the 500ms stub. A spinner or "validating..." indicator would help.
-   **Caching**: If the same image is validated repeatedly, can we cache the result?
-   **Configurability**: The service URL is hardcoded. Eventually it should be a setting in the extension's options page.
